1. **二叉树后序遍历的非直接实现方式** -- gzh

   ​		**来源：LC.145题解 -- [Edward Elric](https://leetcode.cn/u/zdxiq125/)**

   ​		二叉树前中后序的递归版本属于easy题，而迭代版本通常是medium甚至是hard。

   ​		在做迭代版本之前，我建议大家先问问各类“遍历”算法的本质是什么？是最后输出的那一串有序的数字吗？数字的顺序是对的，遍历算法就是对的吗？

   ​		个人认为，以上问题的答案都应该是：否。“遍历”的本质是对内存的有序访问，失去了访问顺序，即便你用各种数据结构恢复了这个次序，遍历本身也显得毫无意义。常见的后序遍历写法中有一种已经出现在评论区了——它的思想也很简单，大家做过单词串翻转吗？

   ```cmake
   String in  = "it is a word"
   String out = "word a is it"
   ```

   ​		这个问题有一种很优雅的写法是先reverse(in)，然后再逐词翻转——“两次反转”，最后得到的就是正确顺序。

   ​		回到二叉树后序遍历，你也可以利用这种思想，利用双向链表的addFirst，对外部次序和内含次序进行同时翻转，同样会得到一种非常”优雅”的解法，结构简单明晰，甚至还有点好背（狗头）。但是，它并没有真正实现“遍历”——仔细看会发现，该算法其实在使用一种异构的前序遍历：“中->右->左”，而非传统意义上的“中->左->右”，而这种异构也正是他的第一次反转。而第二次反转就在输出序列上。

   ​		所以本质上，这是一个“前序遍历”，而不是所谓的“后序遍历”。只有当你的各个节点以“左->右->中”的次序依次出现在迭代的loop当中时，它才是真正的后序遍历，就像官解那样。贴个我个人的版本。

   ```
   class Solution {
       public List<Integer> postorderTraversal(TreeNode root) {
           List<Integer> ans = new ArrayList<>();
           Stack<TreeNode> s = new Stack<>();
           Set<TreeNode> seen = new HashSet<>();
           while (root != null || !s.isEmpty()) {
               if (root == null && seen.contains(s.peek())) {
                   ans.add(s.pop().val);
               } else if (root == null) {
                   seen.add(s.peek());
                   root = s.peek().right;
               } else {
                   s.push(root);
                   root = root.left;
               }
           }
           return ans;
       }
   }
   ```

3. **标题** -- xkz

   在一般情况下，我们通常用fork配合exec函数使用，并在父进程中wait等待，因此统一讨论

   1.fork()  系统调用，通过复制一个现有的进程来创建一个全新的进程。调用的原进程称为父进程，新进程称为子进程，因此存在两次系统调用的从内核返回，也即返回原进程和子进程

   在创建的过程中，父子进程共享代码段，但无法共享数据段，堆栈，如果在创建时就进行复制，不仅会产生高额的读写性能开销，而且，fork的子进程通常要通过exec调用启动调用程序，这是会覆盖掉与原来的数据提供给新程序。因此如果创建时复制，性能代价大。

   ​	因此使用了写实拷贝的技术，创建后临时指向同一个物理内存页，对该物理内存页实行写保护，一旦由一方因为写保护触发了缺页异常，内核为该页创建一个新的物理页面，此时才由真正的独立页面，并且撤销写保护。

   ​	在这个过程中，哪个进程触发的写保护就为谁分配新物理页，另一个进程不变，但由于写保护的关闭，此物理页的内容以及独立。

   这个过程缺页异常的过程

   1.缺页异常最终调用do_pape_fault

   2.进而调用handle_mm_fault

   3.再调用handle_pte_fault

   4.调用do_wp_pape  该函数会创建副本

   ​		同时也涉及到，父子进程运行先后的问题，这个决断需要结合实际的情况处理

   父进程先运行的好处在于，fork前后，父进程在CPU处于活跃状态，CPU的硬件内存管理单元TLB中，目前缓存的都是父进程的页表项，因此父进程继续执行，处理器性能更高

   子进程先运行的好处在于，如果父进程运行，大概率要触发写保护，进而进行复制页，而子进程创建后通常很快就调用exec了，如果父进程先运行，进行了大量复制，而子进程运行后，这些复制页大可能会被完全覆盖掉，如果先调用子函数，能节省这部分潜在的性能开

###### 第一部分：进程基础与虚拟化

**进程 vs 程序：请简述两者的区别。为什么说“安装了没运行”的代码不是进程？**

- **程序**：是一个静态的、存储在磁盘上的可执行文件（如二进制文件、脚本），它只是一系列指令和数据的集合，本身不占用CPU、内存等运行资源。

- **进程**：是程序的一次动态执行实例，是操作系统进行资源分配和调度的基本单位。当程序被加载到内存中，并为其分配了地址空间、堆栈、文件描述符等资源后，才成为一个进程。它有自己的状态、PID等。

   因此，“安装了没运行”的代码只是一堆文件，没有被操作系统加载和执行，没有分配运行资源，所以不是进程。

   **在 Linux 内核中，保存进程信息的结构体叫什么名字？**

   - **`task_struct`**。它包含进程的所有信息：PID、状态、内存映射、文件描述符表、信号处理等。

   ###### 	第二部分：进程状态与调度

   **进程状态：请解释运行态、就绪态、等待（睡眠）态之间的转换逻辑。**

   - **运行态（Running）**：进程正在CPU上执行。
- **就绪态（Ready）**：进程已经具备运行条件，等待被调度器分配CPU。
   - **等待态（Waiting / Sleeping）**：进程因等待某个事件（如I/O、信号、锁）而暂时无法运行。
   
   转换逻辑：

   - 就绪 → 运行：被调度器选中。
- 运行 → 就绪：时间片耗尽或被更高优先级抢占。
   - 运行 → 等待：主动调用阻塞系统调用（如 read、sleep），或等待资源。
   - 等待 → 就绪：等待的事件发生（如I/O完成、信号到达）。
   
   **时间片轮转：为什么 CPU 要划分时间片？如果一个进程一直抢不到时间片会发生什么？**

   - **时间片**：CPU将执行时间划分为小段，轮流分配给各个进程，从而实现多任务并发。这保证了每个进程都能获得一定的CPU时间，避免某个进程长时间独占CPU导致其他进程“饿死”。
   - 如果进程一直抢不到时间片（例如优先级过低），它就会处于**就绪态**，得不到执行，这就是“饥饿”现象。长时间饥饿可能导致进程得不到响应，但在现代调度算法中通常会防止绝对饥饿（如动态提升优先级）。

   **调度策略：现代操作系统的调度策略通常是单一的吗？“会哭的孩子有奶吃”在优先级（NI/PRI）中是如何体现的？**

   - 现代操作系统通常采用**多级调度策略**，结合了优先级、时间片、动态调整等。例如Linux的**CFS（完全公平调度器）**。
- **“会哭的孩子有奶吃”** 在优先级中体现为：进程可以通过调整**nice值（NI）** 来影响其动态优先级（PRI）。nice值越小，优先级越高，越容易获得CPU时间。一个进程如果频繁使用CPU（即“哭”），其动态优先级可能会被降低；反之，如果长期等待，优先级可能会提升，这就是一种“动态调整”，保证公平性。

   ###### 第三部分：监控与管理命令

   **ps -elf 和 ps aux 的区别是什么？**

   - **`ps -elf`**：使用标准Unix风格选项，显示完整信息（包括UID、PID、PPID、C（CPU使用率）、PRI、NI、SZ、WCHAN等）。
- **`ps aux`**：使用BSD风格选项，显示用户、PID、CPU%、MEM%、VSZ、RSS、TTY、STAT、START、TIME、COMMAND。
   - 两者都显示所有进程，只是输出格式和字段不同，`ps -elf`更适合查看父进程、优先级等信息，`ps aux`更简洁。
   
   **free -h 中的 available 和 used 哪个更能反映真实的剩余内存？**

   - **`available`** 更能反映真实的剩余内存。它表示在不触发交换的情况下，系统还可以为进程分配多少内存，包含了可回收的缓存和缓冲区。而 `used` 包含了被使用的内存，但其中一部分是缓存，可被回收。所以 `available` 更贴近实际可用的空闲内存。

   **Swap（交换区） 是干什么用的？它的存在对性能有什么影响？**

   - **Swap**：当物理内存不足时，将部分不常用的内存数据交换到磁盘上的一块区域（swap分区或文件），释放物理内存给其他进程使用。
- **性能影响**：Swap 是磁盘操作，速度远慢于内存，过度使用会导致系统性能急剧下降（称为“swap thrashing”）。因此，Swap 应该作为应急手段，理想情况下物理内存应足够使用。

   **负载 (Load Average)：top 命令里的 1min、5min、15min 平均负载，如果数值远大于 CPU 核心数说明了什么？**

   - **平均负载**：表示单位时间内，系统中处于**运行态和不可中断睡眠态**的进程数。如果数值远大于CPU核心数（比如4核CPU负载>4），说明系统有大量进程在等待CPU，可能发生了过载。但也要注意，负载高也可能是I/O密集型任务（不可中断睡眠）导致的。

   **进程信号：kill -2 和 kill -9 有什么本质区别？**

   - **`kill -2`（SIGINT）**：发送中断信号，相当于按Ctrl+C，进程可以捕获该信号并执行清理后退出，是“温柔劝退”。
- **`kill -9`（SIGKILL）**：发送强制终止信号，进程无法捕获或忽略，内核直接杀死进程，是“强制暴力”，可能导致资源泄漏（如临时文件未删除），应作为最后手段。

   ###### 第四部分：进程控制函数

   **fork() 函数：**

   **fork() 的返回值有哪三种情况？父子进程如何通过返回值“分家”？**

   - **返回值**：
  - 在父进程中，返回子进程的PID（>0）。
     - 在子进程中，返回0。
     - 如果出错，返回-1。
   - 通过判断返回值，父子进程可以执行不同的代码路径，实现“分家”。

   **什么是写时复制 (Copy-on-Write)？它优化了什么性能问题？**

   - **写时复制**：fork 时，父进程的物理内存页并不立即复制给子进程，而是共享同一份内存，并标记为只读。只有当父或子进程试图写入某页时，才真正复制该页给写入方，保持独立性。
- **优化**：避免了 fork 时大量内存复制，提高 fork 效率，节省内存，特别适用于 fork 后立即 exec 的场景。

   **exec 函数族：**

   **execl 和 execv 在传参方式上有什么区别？末尾为什么要加 (char \*)0？**

   - **`execl`**：以**可变参数列表**方式传递参数，每个参数单独列出，最后一个参数后必须跟 `(char *)0` 表示参数列表结束。
  - 例如：`execl("/bin/ls", "ls", "-l", (char *)0);`
   - **`execv`**：以**字符指针数组**方式传递参数，数组的最后一个元素也必须是 `NULL`。
     - 例如：`char *argv[] = {"ls", "-l", NULL}; execv("/bin/ls", argv);`
   - 末尾加 `(char *)0` 或 `NULL` 是为了告诉 exec 函数参数结束，因为它依赖于这个结束标记来遍历参数列表。

   **特殊进程处理：**

   **孤儿进程：谁会收养它？它其实没有什么危害，只是会换一个父进程收拾资源**

   - 孤儿进程被 **init 进程（PID=1）** 收养。当父进程结束，子进程变成孤儿，init 进程会定期调用 `wait` 回收它们，没有危害。

   **僵尸进程：它是怎么产生的？**

   - 僵尸进程产生于子进程退出后，父进程尚未调用 `wait` 或 `waitpid` 获取子进程的终止状态。此时子进程的进程描述符仍然保留，成为一个“僵尸”。如果父进程一直不调用 wait，僵尸会一直存在，消耗少量资源。

   **wait/waitpid：如何利用 waitpid 的 WNOHANG 参数实现非阻塞等待？**

   - `waitpid(pid, &status, WNOHANG)` 不会阻塞进程，如果子进程尚未退出，它立即返回0；如果子进程已退出，则返回子进程PID。可以通过循环调用，实现非阻塞检查。

   ###### 第五部分：进程组、守护进程与日志

   **进程组与会话：**

   **终端关闭时，为什么前台进程通常会死掉？如何让进程脱离终端运行？**

   - 当终端关闭，内核会向该终端关联的**前台进程组**中的所有进程发送 **SIGHUP** 信号，默认行为是终止进程。
- 要脱离终端，可以：
     - 使用 `nohup` 命令启动进程。
     - 在进程中调用 `setsid()` 创建新会话，脱离原终端。
     - 使用 `&` 将进程放入后台，并重定向标准输入输出。
     
      
   
   **守护进程 (Daemon)：**
   
   **创建一个守护进程的必要步骤有哪些？**
   
   1. **fork** 并让父进程退出，使子进程成为孤儿（被 init 收养）。
2. **setsid** 创建新会话，脱离终端。
   3. 改变工作目录为 `/`，避免占用可卸载文件系统。
   4. 重设文件权限掩码（`umask(0)`），避免继承父进程的掩码。
   5. 关闭不再需要的文件描述符（0、1、2 等），并重定向到 `/dev/null` 或日志文件。
   6. 处理信号（可选）。
   
   **日志管理：**

   **为什么生产环境不建议直接 printf 到屏幕？**

   - 屏幕输出（stdout）性能差，会阻塞进程；且屏幕日志无法持久化，不便排查；此外，守护进程通常无终端，输出会失败。

   **日志级别（DEBUG, INFO, WARN, ERROR）的作用是什么？如何实现“只看错误日志”？**

   - **日志级别**：用于分类日志重要程度，便于过滤和排查。从低到高依次为 DEBUG、INFO、WARN、ERROR、FATAL。

   - **只看错误日志**：可通过设置日志输出级别为 ERROR，只输出该级别及以上日志。在程序中可配置日志系统（如 syslog、log4c）的阈值。

   ###### 第六部分：进程间通信 (IPC)

   **管道：匿名管道和有名管道 (FIFO) 的适用场景有什么区别？**

   - **匿名管道**：只能在**有亲缘关系的进程**（父子、兄弟）之间使用，由 `pipe` 创建，通过文件描述符传递。
- **有名管道（FIFO）**：可以在**任意两个进程**之间通信，通过文件系统中的一个特殊文件进行，使用 `mkfifo` 创建，通过路径名打开。

   **共享内存：为什么它是所有 IPC 方式中最快的？**

   - 共享内存将同一块物理内存映射到多个进程的虚拟地址空间，进程间通信**不需要内核参与数据复制**，只需在映射后直接读写内存，因此速度最快。

   **其他方式：除了管道和共享内存，还有哪些进程间通信的手段？**

   - 信号（Signal）
- 消息队列（Message Queue）
   - 信号量（Semaphore）——用于同步而非数据传输
   - 套接字（Socket）——用于网络或本地通信
   - 内存映射文件（mmap）
   - 文件（File）——通过磁盘文件交换数据，但效率低

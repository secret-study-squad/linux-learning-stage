1. **fork** -- gzh

   - **内存复制**

     调用fork函数，fork函数会创建一个和当前进程拥有着几乎一致的用户态地址空间, 作为新进程的地址空间。

     尽管子进程是父进程的副本，但某些属性和资源是根据定义或需要在子进程中进行修改的(主要是PCB)，以确保两个进程是独立的。

   - **写时复制**

     为了减少数据复制的开销, 优化内存管理, fork采用是**写时复制（Copy-On-Write，简称COW）的策略**。

   - **文件打开和fork**

     当使用fork()函数创建子进程**之前**我们打开了一个文件，fork时候子进程会拷贝父进程的文件描述符数组，文件描述符数组中存储的指向一个**文件表项**的指针。但是需要注意的是文件表项是操作系统维护的、可以多个进程程共享。这意味着父进程和子进程将共享打开的文件表项。而在这个**文件表项**中, 不仅维护了指向磁盘文件的inode指针, 还维护有文件读写位置的偏移量。这就意味着, 父子进程对同一个文件对象会共享读写位置。（前提是fork前已经打开open文件生成了文件描述符fd）

   - **补充**

     fd → 文件对象（文件表项）→ inode → 磁盘块
     文件表项即文件对象即struct file 结构体

3. **LINUX的进程调度** -- xkz

   Linux内核存在一个专门用来调度进程的内核组件，称为`调度器`, 调度器的基本工作是从一组处于`可执行状态的进程`中选择一个来执行。

   和很多其他操作系统一样， Linux提供了`抢占式`的多任务模式。调度器会决定某个进程在什么时候停止运行，并且可以让另一个进程进入执行，我们称这个`进程在CPU上替换执行的行为`为`抢占`。

   一些常见发生`抢占CPU`的情况:

   1.时间片用尽；

   2.更高优先级的进程变为可运行状态；

   3.中断和系统调用

7. **标题** -- ywj

   **==fork()会复制什么？==**

   **·用户态：FILE 结构体——会被复制**
   由于 FILE 结构体存在于用户空间的堆区或全局区，而 fork() 的核心逻辑是复制整个用户态地址空间（采用 Copy-On-Write 技术）：

   物理上的复制： 子进程会获得父进程 FILE 结构体的一个精确副本。

   缓冲区的陷阱： 如果父进程的 FILE 缓冲区里还有没刷出的数据（比如你 printf 了一个字符串但没加 \n），fork 之后，父子进程的缓冲区里都会有这份数据。

   后果： 如果不注意，父子进程在各自退出时都会刷出缓冲区，导致同一行日志在文件里打印了两次。这就是为什么在 fork 之前通常要执行 fflush(NULL)。

   **内核态：文件对象 (struct file)——不会复制，而是共享**
   内核里的文件对象属于系统级资源，不属于任何一个特定的进程地址空间：

   引用计数增加： fork() 时，内核会复制父进程的 文件描述符表 (FD Table)。子进程表里的 fd 指向的依然是父进程那个相同的内核文件对象。

   增加计数： 内核会将该文件对象的 f_count（引用计数）加 1。

   共享偏移量： 因为父子进程指向同一个内核文件对象，而“当前读取位置 (Offset)”是存在文件对象里的，所以它们共享同一个偏移量。

   父进程读了 10 字节，子进程接着读就是从第 11 字节开始。
